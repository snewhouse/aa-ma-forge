## Where to level up (specific, anchored)

**Schema-extension test sweep should be a habit.** AD-V2-016 caught a stale `@pytest.mark.slow` test 6 hours after M2c.4 added the `level` field to the overlap pair contract. The test ran only on explicit opt-in (`pytest -m slow`), so the default 414/414 pass at M2c and 420/420 at M3 both skipped it. Lesson: when extending a structured-output contract, `git grep -F` the OLD shape across the entire test directory in the same change set, regardless of test-marker gating. This is L-080 (sync Result Logs) applied to test contracts.

**Status flags drifted on M1.5 and M2c.2.** Both had completed work and full Result Logs but `Status: PENDING` flags that weren't flipped at the time the work shipped. Caught at M4.3 by the L-081 sub-step consistency check (`grep -c "Status: PENDING"`). The fix is mechanical (flip flag) but the prevention is structural: when an agent writes a Result Log, the same edit MUST flip the status flag. The skill prompt should treat them as one operation, not two.

**3 sets of duplicate stale stubs cleaned at M2c.1 and M4.3.** Plan-amendment leftovers: when a plan section is rewritten mid-flight, the OLD PENDING stub stays around because the editor adds the new content but doesn't delete the old. M2a had 2 stale stubs (cleaned at M2c.1); M2c had 3 (cleaned at M4.3). This is a quieter version of the schema-extension drift: when amending a plan section, search the section for OLD entries and remove them in the same change.

**The "deferred sweep" pattern at M2c.6 worked, but the boundary should have been clearer at plan time.** The original plan AC for M2c.6 called for a 4-budget × 2-repo × 3-run sweep (40 cells). At M2c.6 we narrowed it to single-budget aa-ma-forge smoke (5 cells) and deferred the full sweep to M3 prep (AD-V2-014). The full sweep then took ~45 min of wall-clock at M3, which would have been ~45 min of M2c overrun if it had stayed in M2c. The amendment was correct; the original AC was over-scoped. Lesson: when an AC includes "run a multi-hour sweep to validate," ask at plan time whether the sweep is the milestone deliverable or the next milestone's input data.

## 3 habits for next plan

1. **Test-sweep on schema extensions.** Before committing an output-contract extension, `git grep -F` the OLD key set across the entire test directory. Include `@pytest.mark.slow` and other marker-gated tests. Update every match in the same change set.
2. **Status-flag flip on Result Log write.** Treat "write Result Log" and "flip Status from PENDING to COMPLETE" as one atomic operation, not two. The aa-ma-execution skill should enforce this in its sub-step gate.
3. **Multi-hour sweeps belong to the consumer milestone, not the producer.** When an AC includes a sweep-to-validate, ask at plan time: is the sweep itself the deliverable, or is it data input for the next milestone? Default the sweep to the consumer milestone.
