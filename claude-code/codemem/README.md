# codemem

Git-history-aware code intelligence for Claude Code. Pure Python + SQLite. No embeddings, no graph DB, no cloud.

codemem parses your repo, indexes symbols and call edges into a local SQLite database, and exposes 12 MCP tools so Claude Code can answer questions like *"who calls this?"*, *"what changes together with `CLAUDE.md`?"*, and *"show me the context for this AA-MA task."* A fresh build on a 70-file repo takes 0.2 seconds.

This README is the reference. If something here diverges from the code, the code wins — file an issue.

## Quick start

```bash
# 1. Install the plugin directory (symlinks into ~/.claude/, no network calls).
bash /path/to/aa-ma-forge/scripts/install.sh

# 2. Drop a project-scope MCP config into your target repo.
cat > <your-repo>/.mcp.json <<'EOF'
{
  "mcpServers": {
    "codemem": {
      "command": "uv",
      "args": ["--directory", "${PWD}", "run", "python",
               "/path/to/aa-ma-forge/claude-code/codemem/mcp/server.py"],
      "env": {
        "CODEMEM_REPO_ROOT": "${PWD}",
        "CODEMEM_DB": "${PWD}/.codemem/index.db"
      }
    }
  }
}
EOF

# 3. Open a fresh Claude Code session in that repo.
#    Approve when prompted. The first tool call triggers an auto-build.
```

Alternative — user-scope install (no per-repo `.mcp.json`):

```bash
claude mcp add --scope user codemem uv \
  --directory /path/to/aa-ma-forge \
  run python claude-code/codemem/mcp/server.py
```

After install, `/mcp` in a fresh session lists `codemem` as connected.

## The 12 MCP tools

Six are ports of Eric Buess's `/index` tools with the same names and closely-related semantics. Six are new.

### Ported from `/index` (M1)

| Tool | What it answers |
|------|-----------------|
| `who_calls(name, max_depth=3)` | Upstream callers of a symbol. Also registered as alias `find_references`. |
| `blast_radius(name, max_depth=3)` | Downstream transitive callees — what breaks if you change `name`. |
| `dead_code()` | Function and method symbols with zero incoming call edges. Also `find_dead_code`. |
| `dependency_chain(source, target, max_depth=5)` | Shortest call-graph path from `source` to `target`. *Semantics differ from `/index` — see [migration guide](../../docs/codemem/migration-from-index.md).* |
| `search_symbols(query)` | Substring match, ranked exact > prefix > contains. *`/index` uses regex; codemem uses substring.* |
| `file_summary(path)` | Symbols defined in a file, ordered by source line. |

### Git mining + AA-MA-native (M3)

These are the reason codemem exists.

| Tool | What it answers |
|------|-----------------|
| `hot_spots(window_days=90, top_n=10)` | Top-N files ranked by `(commits in last N days) × (function count)`. |
| `co_changes(file_path, threshold=3)` | Files that change in the same commits as `file_path` **and have no import edge to it.** Surfaces hidden coupling. |
| `owners(path, refresh=False)` | Per-author line-count percentages from `git blame`. Works on files or directories. |
| `symbol_history(name, file_path=None)` | `git log -L:<name>:<file>` summary: first-seen, last-touched, authors, change count. |
| `layers()` | Three-tier ASCII onion (core / middle / periphery) grouped by in-degree. Fits 80-column terminals. |
| `aa_ma_context(task_name)` | For an AA-MA task: filtered hot-spots, owners of mentioned files, blast-radius of mentioned symbols. Opt-in write mode with `--write` appends to `<task>-reference.md`. |

Every tool takes a `budget` parameter (default 8000 tokens). Output over budget gets truncated with `truncated: true` in the response. No streaming; no unbounded payloads back into the context window.

Every tool argument that names a symbol or path is sanitised before it reaches SQL or `subprocess`: allow-list regex `^[A-Za-z0-9_./\-]{1,256}$` for symbol args, plus `Path.resolve` + `relative_to(repo_root)` for paths. Adversarial inputs return a structured error dict — see [`SECURITY.md`](../../SECURITY.md).

## What's different — the landscape, fairly

There's a growing ecosystem of MCP-based code-intelligence servers. Each one makes different choices. None of the tools below are "worse" than codemem; they're pointed at different questions.

### [`/index`](https://github.com/ericbuess/claude-code-project-index) (Eric Buess)

The direct predecessor. Six MCP tools, dense JSON output, regenerated by a Stop hook on every session end. codemem ports those six tools with small signature changes and adds six more. If `/index` fits your workflow, keep using it — codemem was built for the cases where git-history + AA-MA context matter. Full migration walkthrough in [`docs/codemem/migration-from-index.md`](../../docs/codemem/migration-from-index.md).

### [GitNexus](https://github.com/abhigyanpatwari/GitNexus) (~27k stars)

Tree-sitter call graph + MCP, with tool names like `impact`, `context`, `detect`. Large, mature, fast-growing. Different naming convention (we register `find_references` / `find_dead_code` as aliases so agent tool-search reaches codemem when hunting for the GitNexus names). GitNexus users have pushed back on the tool writing `CLAUDE.md`/`AGENTS.md` into their repos uninvited; codemem never writes to your repo without `--write` passed explicitly.

What codemem has that GitNexus doesn't (as of April 2026): git-mining tools (`hot_spots`, `co_changes`, `owners`, `symbol_history`, `layers`) and AA-MA coupling (`aa_ma_context`).

### [Axon](https://github.com/harshkedia177/axon) / [Axon.MCP.Server](https://github.com/ali-kamali/Axon.MCP.Server)

Graph-powered code intelligence engines that index into a knowledge graph and expose MCP tools. The ali-kamali build adds vector-powered semantic search via embeddings. If you want semantic "find me code that does X-ish" queries, Axon is probably a better fit than codemem. codemem deliberately does NOT use embeddings — symbol names + call edges + git history are enough for the questions we care about, and skipping embeddings keeps the whole system pure-Python, no GPU, no model downloads, no cloud dependencies.

### [CodeGraph](https://github.com/nahisaho/CodeGraphMCPServer) / [CodeGraphContext](https://github.com/CodeGraphContext/CodeGraphContext) / [colbymchenry/codegraph](https://github.com/colbymchenry/codegraph)

Several independent projects, most using a graph database (Neo4j or similar) for storage and GraphRAG for queries. Similar tradeoff to Axon: graph DB + retrieval-augmented generation trades pure-Python simplicity for richer semantic queries. codemem's SQLite + recursive-CTE call graph is simpler to deploy (no external DB daemon) but will lose to these tools on "find me code semantically related to this comment" queries.

### [jCodeMunch](https://github.com/jgravelle/jcodemunch-mcp) (J. Gravelle)

Tree-sitter AST parsing with aggressive token-efficient retrieval — they benchmark a 95% average token reduction vs brute file reads, with one FastAPI case at 99.8%. Focused tightly on "fetch code by symbol without reading the whole file." codemem also budget-caps output (default 8000 tokens, binary-search truncation) but doesn't currently match jCodeMunch's retrieval-ratio benchmarks. If pure token-efficiency for code retrieval is your primary constraint, measure both.

### The short version

- **codemem**: git-history as a first-class signal, AA-MA coupling, SQLite canonical, no embeddings. Best when you want `co_changes`, `hot_spots`, `owners`, or AA-MA workflow integration.
- **`/index`**: simpler scope (6 tools, one JSON file), direct predecessor, still excellent for its chosen scope.
- **GitNexus**: broader adoption, different tool vocabulary, tree-sitter engine.
- **Axon / CodeGraph**: graph DB + embeddings. Best for semantic queries.
- **jCodeMunch**: best-in-class retrieval token-efficiency.

You can run more than one MCP server at a time. Claude Code routes by tool name. If GitNexus is already serving `impact` and codemem is serving `blast_radius`, both work — the agent picks based on the question.

## Install — the longer version

### Prerequisites

- Python 3.13+ (3.11+ supported; 3.13 is what we test against)
- `uv` package manager (`curl -LsSf https://astral.sh/uv/install.sh | sh`)
- `git`
- `ast-grep` — shipped inside the wheel, no separate install needed for non-Python languages

### Option A — project-scope `.mcp.json` (recommended)

Described in [Quick start](#quick-start) above. The `.mcp.json` lives at your target repo's root and ships with your project. First fresh session prompts for approval; `claude mcp reset-project-choices` resets those approvals if you need to re-approve.

This is the right choice when you want codemem enabled per-repo rather than globally.

### Option B — user-scope via `claude mcp add`

```bash
claude mcp add --scope user codemem uv \
  --directory /path/to/aa-ma-forge \
  run python claude-code/codemem/mcp/server.py
```

Writes to `~/.claude.json` → top-level `mcpServers.codemem`. Loaded for every Claude Code session regardless of which repo you're in. Environment variables `CODEMEM_REPO_ROOT` and `CODEMEM_DB` default to `$PWD/.codemem/index.db` if unset, so the server uses whichever repo you happen to be working in.

### The post-commit hook (opt-in)

`scripts/install.sh --wire-git-hook` appends a background-refresh line to your repo's `.git/hooks/post-commit`. After this, every `git commit` triggers `codemem refresh` in the background; Claude Code sees an up-to-date index without manually running `codemem build`.

Not installed by default — modifying `.git/hooks/` in a repo that isn't aa-ma-forge's is opt-in. Disable later by deleting the sentinel-marked line from the hook file.

Auto-build on first MCP query (delivered by M3.5 Task 3.5.3) means the hook isn't strictly necessary — `.codemem/index.db` will be built the first time Claude Code queries any codemem tool. The hook is useful when you want queries to be sub-second instead of waiting for a cold build.

### What gets written where

- `<repo>/.codemem/index.db` — SQLite database. Git-ignored by default.
- `<repo>/.codemem/wal.jsonl` — codemem's JSONL write-ahead log for crash-safe incremental refresh. Rotates at 10 MB with 3 compressed archives retained.
- `<repo>/.codemem/db.lock` — single-writer advisory lock. Harmless if left over from a crashed indexer.
- `<repo>/.codemem/refresh.pid` — PID of the most-recent post-commit refresh. Only exists if the git hook is installed.
- `<repo>/.codemem/refresh.log` — structured log line per refresh invocation.
- `<repo>/PROJECT_INTEL.json` *(optional)* — PageRank-ranked 1024-token symbol index. Commit this one if you want it.

codemem never writes outside `<repo>/.codemem/` or `<repo>/PROJECT_INTEL.json` — see [`SECURITY.md`](../../SECURITY.md) for the full contract.

## Performance

On aa-ma-forge itself (70 tracked files, ~10k LOC Python + a bit of shell):

- Cold build: **0.172s** median across 5 trials (vs `/index` at 0.235s — 0.73× ratio).
- Warm build (re-run with unchanged files): **0.18s**.
- `who_calls("extract_python_signatures")` on `/index/scripts/index_utils.py` (anchor query): **~0.27ms** via the in-memory MCP handler (CLI cold-start adds ~45ms of sqlite-open overhead on top).
- `PROJECT_INTEL.json` at `--budget=1024`: stays under 5 KB.

All budgets enforced by `tests/perf/test_budgets.py` — CI fails if any regresses. Full methodology and raw numbers: [`docs/codemem/performance-slo.md`](../../docs/codemem/performance-slo.md). Comparison against `/index`: [`docs/benchmarks/codemem-vs-index.md`](../../docs/benchmarks/codemem-vs-index.md).

Medium-repo (~50k LOC) and large-repo benchmarks are pending user-provided reference repos — see the benchmark file for how to reproduce.

## Architecture in one paragraph

Four-layer cake: the public tool surface in `codemem.mcp_tools` talks to `codemem.indexer` / `codemem.resolver` / `codemem.pagerank`, which in turn use `codemem.parser` (stdlib `ast` for Python, `ast-grep` subprocess for everything else), which writes through `codemem.storage` (SQLite + the WAL JSONL journal). Import-linter enforces that upper layers never skip layers and the parser stays pure. The MCP server at `claude-code/codemem/mcp/server.py` is a thin adapter — it never imports the lower layers directly, only `codemem.mcp_tools`. Full detail: [`docs/codemem/ARCHITECTURE.md`](../../docs/codemem/ARCHITECTURE.md).

## CLI

```bash
codemem build                   # full index
codemem refresh                 # incremental refresh since last SHA
codemem status                  # counts + DB path + user_version
codemem query <tool> [args...]  # invoke any of the 6 M1 tools directly
codemem intel                   # write PROJECT_INTEL.json
codemem replay --from-wal       # rebuild DB from the JSONL WAL
```

`codemem --help` enumerates the full surface. The CLI and the MCP server share the same underlying `codemem.mcp_tools.*` functions — no behaviour divergence.

## What could make us abandon this

Five measurable signals would cause us to kill or pivot codemem. Thresholds, rationale, and current status are in [`docs/codemem/kill-criteria.md`](../../docs/codemem/kill-criteria.md). As of April 2026, none have fired — the M1 speed gate cleared at 0.73× `/index` on the small reference repo, the M2 crash-recovery tests are deterministic on CI, and no competing tool has shipped comparable git-mining coverage. The one to watch is the moat-evaporated signal (GitNexus or Axon shipping three or more of our git-mining tools with comparable UX).

We'd rather tell you up front what would make us walk away than have you find out from a stale repo.
